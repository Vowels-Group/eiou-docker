# Changelog

All notable changes to the eIOU Docker project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

This project does not yet follow [Semantic Versioning](https://semver.org/). There are
no version tags. Entries are organized by development period rather than release number.

## [Unreleased]

### Changed
- **Advanced Settings categories expanded with `settings-group-heading` subsections for scannability**: the Feature-Toggles-only subsection treatment introduced earlier now extends to every dropdown-accessed category with more than ~3 fields. **Main Wallet Settings** (previously a flat 10-field grid) now has **Appearance** (contact avatar style, amount/status color schemes), **Identity** (display name), **Payments** (default currency, fees, credit limit), and **Network** (default transport mode). **Advanced → Backup & Logging** splits into **Backup** (time, retention count) and **Logging** (max entries, log level). **Advanced → Data Retention** formalizes its existing soft split: the cleanup block is now headed **Cleanup** and the archive block **Archive** (the previous `h4` "Archive Retention (Days)" heading was inconsistent with the Feature-Toggles `h5` `settings-group-heading` style). **Advanced → Rate Limiting** splits into **Throughput** (P2P per-minute cap) and **Attempt Blocking** (max attempts, window, block duration) — matching the "two independent mechanisms" language already in the intro prose. **Advanced → Network** (9 fields, the longest category) splits into **Transport Timeouts** (HTTP/Tor), **Tor Resilience** (circuit max failures, cooldown, fallback toggles), **Routing & Delivery** (max P2P level, P2P expiration, tx delivery expiry), and **API** (CORS origins). Small categories (Currency, Display, Sync, GUI Security, Reset to Defaults) keep their flat layout — splitting 1-3 fields would be churn, not clarity
- **`.settings-group-heading` no longer `text-transform: uppercase`**: the Feature Toggles subsection headings were rendering as `CONTACTS` / `TRANSACTIONS` / `GUI` / `SYSTEM`, which was inconsistent with the Title Case used by every other heading in the settings page (and indeed every subsection heading elsewhere in the GUI). Removed the `text-transform` and the paired `letter-spacing: 0.02em` so the HTML content ("Contacts", "Transactions", …) renders as-written. Font-size bumped from `0.9rem` to `0.95rem` to match the visual weight that the uppercase + letter-spacing previously provided
- **`.settings-group-heading` divider rule switched from `:first-of-type` to `.settings-grid + .settings-group-heading`**: the old rule zeroed out the top border + padding for any h5 that was the first `settings-group-heading` among its siblings. That worked for Feature Toggles (where "Contacts" sits right under the section h4 and shouldn't carry a divider) but broke GUI Security (where "Active Remembered Sessions" is the *only* h5 in the section but follows a full settings-grid — the divider was visually needed to separate it from the field row above). New selector moves the border/spacing to an adjacent-sibling match: a heading gets the full hair-line divider + 2.25rem top margin only when it follows a preceding `.settings-grid`. First heading in any category (right under the section h4 or intro prose) stays tight; second-and-later headings in multi-group sections get the divider automatically; and the single-h5-after-grid case is now handled correctly
- **Active Remembered Sessions heading uses `settings-group-heading`** (was a bespoke `active-sessions-heading` class): picks up the new cross-category consistency — same hair-line top border, same typography, same spacing as every other subsection heading in the settings page. The `.active-sessions-container` wrapper no longer carries its own top-border (that treatment is now owned by the heading itself), and the unused `.active-sessions-heading` CSS rule is removed
- **Active Remembered Sessions empty state uses the shared dashed-border `.empty-panel`** (was a plain `<p class="form-hint">` with an info icon): now matches the API Keys empty state's visual treatment — dashed border, centered text, padded panel — so "no remembered browsers yet" reads as an empty-list placeholder rather than an inline help hint. New `.empty-panel` CSS class is defined alongside the existing `.api-keys-empty` rule (shared selectors) so both surfaces get the same treatment without renaming the existing class

### Fixed
- **Failed Messages search bar and status filter now hide when the queue is empty** (matching Contacts and Payment Requests): previously the search input and the "Any status" dropdown rendered even when `$dlqItems` was empty, giving users controls with nothing to act on. Both are now gated on `!empty($dlqItems)` — an empty queue shows just the big-icon empty-state panel and nothing else

### Added
- **Transactions archival with chain-checkpoint invariant**: the `transactions` table no longer grows without bound, and the chain-integrity walk that runs on every outbound send (`TransactionChainRepository::verifyChainIntegrity`) no longer scales with total history. Completed transactions older than `transactionsArchiveRetentionDays` (default 30) move nightly to a new `transactions_archive` table via `transaction-archive-cron.php` at 01:30 UTC (offset 30m from the payment-request archival so the two don't contend for the shared archive-backup artefact). Archival is gated per bilateral pair on a gap-free chain-integrity check — pairs with a detected gap are **skipped**, not archived, so the gap stays inspectable. Clean pairs get a row upserted in a new `transaction_chain_checkpoints` table recording the gap-free-at-archival proof: `archived_count`, `archived_txid_hash` (SHA-256 over the sorted archived-txid list — tamper detection), `highest_archived_timestamp`, `highest_archived_time`, `last_verified_gap_free_at`. Checkpoints are keyed by the canonical (LEAST, GREATEST) of the pair's pubkey hashes so direction doesn't matter. Schema version bumps 9 → 10 so `runMigrations()` creates both new tables on existing databases. `BackupService::ARCHIVE_TABLES` extends to cover `transactions_archive`, so the hot/cold backup split already in place for payment requests works for transactions too.

  **Hot-path optimization**: `verifyChainIntegrityByHashes()` now defaults to a live-only walk plus a single indexed lookup on the pair's checkpoint. If a checkpoint exists, a missing `previous_txid` from live is trusted as an archived row rather than flagged as a gap — collapsing verify from O(all history) to O(live tail) + 1 row. A new optional `$useCheckpoint` parameter (default `true`) lets callers force a paranoid walk that reads both tables end-to-end. The `transaction_count` field reflects live + `checkpoint.archived_count` so totals still represent the whole chain.

  **`eiou verify-chain`** is a new top-level CLI command that audits every bilateral chain on the node with `useCheckpoint=false` AND recomputes each pair's `archived_txid_hash` from the current archive rows, comparing against the stored checkpoint value. Per-pair stdout shows chain status (OK / FAIL with gap detail) + checkpoint hash match/mismatch. Exit 0 clean, 1 on any finding. Intentionally NOT on any cron — O(all history) per pair, run deliberately. Catches tampering on the archive table that the hot path trusts past by design.

  **Read paths extended to span live + archive** for correctness after archival:
  - `getChainStateSummary()` UNIONs both tables so sync negotiation between nodes with different archival states still agree on the txid set (the checkpoint shortcut can't be used here — sync needs the full list).
  - `getContactBalance()` + `getAllContactBalances()` UNION both sides so per-contact balances survive archival; archive branches fall back to live-only on PDOException.
  - `getSentUserTransactions` / `getReceivedUserTransactions` + the `*Address` variants (used by `eiou history`) issue a second SELECT against the archive and merge + re-sort + re-slice in PHP, so CLI history output stays complete.
  - `getTransactionsByType()` pushes the `$type` filter into SQL and UNIONs archive so the `GET /api/transactions?type=X` endpoint no longer scans the full joined history in memory.
  - `TransactionStatisticsRepository` aggregates (`getTotalCount`, `getCountByType`, `getTypeStatistics`, `getStatisticsByType`, `getStatisticsByStatus`, `getOverallStatistics`) all extend across both tables. `getOverallStatistics` uses a UNION ALL subquery so `COUNT(DISTINCT sender_address)` / `COUNT(DISTINCT receiver_address)` deduplicate naturally across the combined set.

  **Archive-aware sync protocol** (load-bearing for real-world use — without this, archival breaks sync between nodes with asymmetric archival states). Four txid lookup paths now consult both tables:
  - `transactionExistsTxid()` — sync dedup. Short-circuits on live hit, falls through to archive on miss. Keeps a counterparty from re-pushing already-archived txs back into our live.
  - `getByTxid()` — `handleSyncNegotiationRequest` uses this to fetch a tx the remote is missing. Archive-only hits are now returned so we can re-push archived txs to a counterparty that doesn't have them.
  - `getStatusByTxid()` — `MessageService::handleTransactionStatusRequest` now answers `completed` from the archive rather than `TransactionNotFound`, so peers don't think an archived tx evaporated.
  - `getLocalTransactionByPreviousTxid()` — chain-conflict detection. A remote tx claiming `previous_txid = X` where X is archived is still a conflict that needs resolution; the archive-hit row is tagged with `_source='archive'` so downstream conflict resolution can apply the archive-wins rule.

  **Archive-wins rule for chain conflicts**: if a chain conflict is detected and the local partner is tagged `_source='archive'`, `resolveChainConflict` forces `winner=local` regardless of lexicographic txid comparison. The remote tx is still inserted into live (both have valid signatures; chain ordering resolves at read time via the deterministic tiebreak), but the `resignLocalTransaction` path is skipped — the archive is never modified. A `WARNING` log entry captures the occurrence. The edge case shouldn't happen under normal operation (archival moves only `completed` rows; chain conflicts happen between `pending` txs) but the rule makes stale-pending / misconfigured-counterparty / adversarial scenarios safe.

  **Settings** exposed through the three standard surfaces: GUI field under Data Retention → Archive (inline note about the chain-integrity gating), API key `transactions_archive_retention_days`, CLI `eiou changesettings transactionsArchiveRetentionDays N` + interactive menu entry 35b + appears in `viewsettings`.

  **Unit tests** cover the invariant (gap-free pair archived + checkpointed; gap pair skipped; mixed-pair case archives only the clean ones; hot-loop protection + backup side-effects), the checkpoint-trust fast path vs paranoid audit path, archive-aware txid lookups (short-circuit on live, fallthrough on miss, archive-missing fallback), chain-audit findings (clean / gap / tampered hash / no checkpoint), and the archive-wins rule with a remote tx whose txid would ordinarily win lexicographically. **Live validation** on the 2-node docker harness: Alice archives 5 txs, Bob still has all 5 in his live table; Alice sends 1 new tx + syncs — Alice's live grows by exactly 1 (not 6), zero duplicates between live and archive, `verify-chain` reports 7 settled (2 live + 5 archive), chain OK, hash matches; tampering the archive flips `verify-chain` to exit 1 with the expected findings.

  Scope deliberately deferred: benchmarks at 10k / 100k / 1M bilateral txs — would require seeding that many real transactions to exercise meaningfully; the O-cost change is visible by code inspection.

### Changed
- **Feature Toggles organised into four subsections**: the flat list of ~15 feature toggles in Advanced Settings was hard to scan. Now broken into **Contacts** (Contact Status, Sync Status on Ping, Auto-Accept Restored Contacts, Auto-Reject Unknown Currencies), **Transactions** (Auto-Accept P2P Transactions, Hop Budget Randomization, Auto Tx Drop Propose / Accept / Accept Guard), **GUI** (Auto-Refresh Transactions, Hide Empty GUI Sections), and **System** (API Server, Auto-Backup, Update Notifications, Anonymous Analytics). ### Fixed
- **Docker Hub image now pulls on linux/arm64 (Apple Silicon, Raspberry Pi, AWS Graviton, etc.)**: the `Publish Docker Image` GitHub Actions workflow now builds + pushes a multi-architecture manifest (`linux/amd64` + `linux/arm64`) instead of an amd64-only image. Previously an arm64 host got `no matching manifest for linux/arm64/v8 in the manifest list entries: not found` when trying `docker pull eiou/eiou:latest`. Build uses native GitHub-hosted runners per arch (`ubuntu-latest` for amd64, `ubuntu-24.04-arm` for arm64 — no QEMU emulation), each pushing by digest, then a merge job assembles the manifest list with `docker buildx imagetools create`. Release assets are now published per-arch: `eiou-node-<version>-amd64.tar.gz` and `eiou-node-<version>-arm64.tar.gz` (users pick whichever matches their host). Provenance attestations are off for now — can be turned back on in a focused follow-up if supply-chain verification becomes a requirement

### Security
- **Hard-require secp256k1 — removed silent fallback to prime256v1**: `BIP39::getPreferredCurve()` previously returned `prime256v1` with only an `error_log` warning when secp256k1 was missing from the linked OpenSSL. - **DLQ transaction retry stuck in "retrying" + never resolves**: three bugs conspired after the refresh-payload feature landed so a transaction DLQ retry could never complete. (1) `DeadLetterQueueRepository::getById` auto-decodes the `payload` JSON column into an array before returning; the refresh code was calling `json_decode` on what was already an array (always returning `null`), silently skipping the refresh, then setting `$item['payload']` back to a JSON string which broke `DlqController`'s array-typed callback with a TypeError on the next retry. (2) `CliDlqService`'s retry callback was typed `function (string $payload, ...)` but `getById` returns an array — the pre-refactor inline code wasn't type-hinted so the mismatch was silent, but my refactor made it explicit and broke CLI retries. - **Sticky table headers transparent on scroll**: the `thead th` elements in the Contacts, Recent Transactions, and Failed Messages tables used `background: inherit` which doesn't paint correctly with `position: sticky`, letting table rows show through the header when scrolling. Replaced with an explicit background color

### Added
- Add **MariaDB Transparent Data Encryption (TDE)** — enabled by default after wallet generation, all database files encrypted at rest using `file_key_management` plugin with key derived from master key via HMAC-SHA256. TDE key stored only in `/dev/shm` (RAM-backed, never persisted to disk).
